Perguntas Frequentes

Respostas diretas e aprofundadas para os principais questionamentos sobre Análise de Sistemas.

O analista de sistemas é o tradutor entre o negócio e a tecnologia. Ele:

  • Levanta requisitos através de entrevistas, workshops e observação.
  • Modela processos com BPMN e especifica sistemas com UML.
  • Acompanha testes, homologação e implantação.
  • Atua como facilitador da comunicação entre stakeholders e equipe técnica.
🔎 Na prática: É a pessoa que garante que o software resolva o problema certo, da maneira certa.

Análise

Define O QUÊ o sistema deve fazer. Foca nos requisitos funcionais e não funcionais.

Projeto

Define COMO o sistema será construído. Escolhe arquitetura, tecnologias e padrões.

Sim, ambos coexistem:

  • Cascata (Waterfall): ideal para projetos com escopo fixo, alta regulamentação (aviação, saúde).
  • Ágil (Scrum, Kanban): melhor quando há incerteza, necessidade de feedback rápido e entregas incrementais.

O analista moderno deve saber atuar nos dois contextos.

Business Process Model and Notation – linguagem visual padronizada para mapear processos de negócio. Usada para:

  • Documentar fluxos atuais (AS-IS).
  • Propor melhorias (TO-BE).
  • Alinhar o entendimento entre áreas de negócio e TI.
💼 Mercado: Conhecimento em BPMN é frequentemente exigido em vagas para analistas de processos e de sistemas.

Não é obrigatório ser um desenvolvedor, mas é fundamental dialogar com a equipe técnica. Conhecimentos valorizados:

SQL Python Java C# JavaScript básico

Compreender lógica, estruturas de dados e integrações (APIs) faz diferença na hora de especificar sistemas.

Funcional: Ação que o sistema executa (ex.: "o sistema deve permitir cadastrar um cliente").

Não funcional: Qualidade do sistema (ex.: "a tela de login deve carregar em menos de 2 segundos").

📋 Exemplo real: Um requisito funcional para um banco: "transferir fundos entre contas". Um não funcional: "a transferência deve ser concluída em até 10 segundos com criptografia TLS 1.3".

  • Integração com sistemas legados: muitos ambientes convivem com mainframes e APIs modernas.
  • Big Data e Analytics: pensar no volume de dados desde a análise.
  • Segurança e LGPD: requisitos de privacidade impactam a modelagem.
  • Cloud e microsserviços: arquiteturas distribuídas exigem nova mentalidade.
  • Ética digital: garantir que o sistema não reproduza vieses ou viole direitos.

O CVDS oferece uma estrutura organizada que reduz riscos e evita retrabalho. Ele garante que:

  • Nenhuma etapa crítica seja ignorada.
  • Os stakeholders tenham visibilidade do progresso.
  • A qualidade seja verificada antes da entrega final.

Mesmo em metodologias ágeis, as fases do ciclo (levantamento, projeto, testes etc.) continuam existindo, apenas de forma iterativa.

  1. Formação: cursos como Análise e Desenvolvimento de Sistemas, Sistemas de Informação ou Ciência da Computação.
  2. Habilidades técnicas: estude modelagem (UML, BPMN), SQL, e conceitos de APIs.
  3. Soft skills: pratique comunicação, escrita de documentos e resolução de conflitos.
  4. Portfólio: documente pequenos projetos de levantamento de requisitos e protótipos.
  5. Certificações: considere ECBA, CBAP ou cursos de Product Owner/Scrum Master.
🌟 Dica: participe de grupos de estudo e contribua em projetos open-source como analista voluntário para ganhar experiência.
