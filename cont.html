1. Definição e Importância
O que é a Análise de Sistemas?

A análise de sistemas não é apenas "escrever código". É um processo sistemático de investigação. O analista atua como uma ponte entre o problema organizacional (o negócio) e a solução tecnológica (o software). O objetivo principal é destrinchar sistemas complexos em partes menores para entender como elas interagem.

🔍 Definição formal: Conjunto de atividades que buscam compreender, modelar e especificar os requisitos de um sistema de informação, garantindo que ele atenda às necessidades dos usuários e da organização.
Por que é tão importante?

Construir um software sem análise prévia é como construir um prédio sem planta arquitetônica. A importância se dá em três frentes:

  • Otimização Redução de retrabalho e eliminação de processos redundantes.
  • Alinhamento Garantir que o TI não crie algo tecnicamente perfeito, mas inútil para o negócio.
  • Economia Erros descobertos na fase de análise custam até 100x menos para serem corrigidos do que na fase de produção.
Exemplo Prático: Um e-commerce está perdendo vendas na tela de pagamento. O analista de sistemas não vai apenas "trocar o botão". Ele mapeia a jornada do usuário, analisa as falhas de integração com a API do banco e propõe um novo fluxo de dados mais rápido e seguro.
2. Evolução dos Sistemas de Informação

A evolução dos sistemas de informação acompanha a própria história da computação. Compreender esse percurso ajuda o analista a contextualizar decisões técnicas e a antecipar tendências.

1. Era dos Mainframes (décadas de 60-70)

Sistemas centralizados, processamento em lote (batch). Usuários não interagiam diretamente. Foco em automação de tarefas repetitivas.

2. Microcomputadores e Redes Locais (anos 80)

Descentralização. Surgimento dos PCs e das redes LAN. Os sistemas passam a ser usados diretamente pelos funcionários.

3. Internet e Sistemas Web (anos 90-2000)

Conectividade global. E-commerce, portais corporativos. A análise de sistemas passa a incluir preocupações com usabilidade e segurança.

4. Cloud Computing e Mobilidade (2010 em diante)

SaaS, IaaS, PaaS. Sistemas acessíveis de qualquer lugar. O analista precisa pensar em escalabilidade, APIs e experiência mobile-first.

💡 Para o analista: Cada geração trouxe novas exigências. O profissional moderno precisa conhecer desde integrações legadas (mainframe) até arquiteturas serverless.
3. Papéis e Responsabilidades do Analista

O analista de sistemas pode atuar com diferentes "chapéus" ao longo de um projeto. Conheça os mais comuns:

Analista de Requisitos

Levanta, documenta e valida as necessidades dos stakeholders. Conduz entrevistas e workshops.

Entregável: ERS
Analista de Processos

Mapeia fluxos de trabalho (AS-IS / TO-BE) usando BPMN. Identifica gargalos e propõe melhorias.

Entregável: Diagrama BPMN
Analista de Dados / Modelador

Define a estrutura de banco de dados, entidades e relacionamentos (MER). Trabalha próximo ao DBA.

Entregável: MER/DER
Analista de Testes (QA)

Planeja casos de teste, executa validações e garante a qualidade antes da homologação.

Entregável: Plano de Testes
🧩 Visão integrada: Muitas vezes o mesmo profissional acumula esses papéis, especialmente em times enxutos. O importante é saber quando cada abordagem é necessária.
4. Ciclo de Vida (CVDS)

O Ciclo de Vida do Desenvolvimento de Sistemas (CVDS) é a estrutura metodológica que guia as equipes desde a ideia inicial até a aposentadoria do software.

1. Levantamento de Requisitos

A fase de "O QUÊ". Entrevistas, questionários e observação para entender exatamente o que o cliente precisa. Entregável: Documento de Especificação de Requisitos.

2. Análise e Projeto (Design)

A fase de "COMO". Desenho da arquitetura, modelagem de banco de dados (MER) e interfaces (Wireframes). Entregável: Protótipos e Diagramas UML.

3. Implementação (Codificação)

Tradução dos modelos para código fonte usando linguagens de programação e frameworks.

4. Testes e Validação

Testes unitários, de integração e de estresse. Busca-se quebrar o sistema antes que o usuário o faça.

5. Implantação e Manutenção

Colocar em produção (Deploy), treinar usuários e realizar correções de bugs ou melhorias contínuas.

5. Metodologias e Abordagens
O Embate Clássico: Cascata vs. Ágil
Cascata (Waterfall)

Abordagem linear e rígida. Uma fase só começa quando a anterior termina. Excelente para projetos com escopo fechado e requisitos imutáveis (ex: software para aviação).

Metodologias Ágeis

Iterativas e incrementais (Scrum, Kanban). Foco em entregas rápidas de valor e adaptação a mudanças. Ideal para startups e ambientes de inovação.

Notações Padrão de Mercado
  • UML (Unified Modeling Language): A linguagem universal para especificar sistemas orientados a objetos. Diagramas de classe, casos de uso e sequência.
  • BPMN: Focado no mapeamento e melhoria contínua dos processos de negócios da empresa.
📌 Dica de ouro: Não existe metodologia "melhor". O bom analista escolhe a abordagem com base no risco, no tamanho do time e na clareza dos requisitos.
